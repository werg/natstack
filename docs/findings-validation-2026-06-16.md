## #1 — vcs.applyEdits path traversal escapes context worktree
- **Verdict:** confirmed  | reported critical → assessed **high**
- **Why:** The finding is real in the current tree. vcs.applyEdits accepts EditOps whose `path` is validated only as `z.string()` (packages/shared/src/serviceSchemas/vcs.ts:50-77) and is reachable from panels/apps/workers/extensions/harness per the service policy (src/server/services/vcsService.ts:171-173, exposed in runtimeSurface.panel.ts:140 and runtimeSurface.worker.ts:72). The handler (vcsService.ts:219-238) forwards `input.edits` unmodified. WorkspaceVcs.applyEdits (src/server/gadVcs/workspaceVcs.ts:1351-1387) keys the file map by `op.path` verbatim, hashes content into the CAS, and ingests it into GAD state with NO rejection of `..`, absolute, or leading-slash components. The path becomes filesystem-unsafe only at materialize time: WorkspaceStore.materializeState computes `const absPath = path.join(dir, ...relPath.split('/'))` (src/server/gadVcs/store.ts:492; the deletion loop at :485 and materializeFileList at :614 do the same), where `dir` is the workspaceRoot or `.contexts/{ctx}` head dir (dirForHead, workspaceVcs.ts:286-290). A `..` component escapes that root. Materialization fires automatically after edits via build-source graph discovery (materializeStateForGraphDiscovery -> materializeState, workspaceVcs.ts:804-818) and on context-fork/merge checkout (lines 842, 1289). The only sanitizer, toVcsPath (workspace/packages/harness/src/tools/tool-vcs.ts:16-24, throws 'Path escapes the workspace root'), runs CLIENT-SIDE inside the untrusted harness sandbox and is entirely bypassed by calling vcs.applyEdits directly. So canonical GAD paths are indeed not validated at the service/server boundary. I adjusted Critical->High only because exploitation requires the writer to run as the server-host OS user and the most-attacker-controllable trigger (build-source materialization) writes into buildSourcesRoot's parent area rather than e.g. system dirs; it is still a sandbox-escaping arbitrary host file write/overwrite/delete.

- **Impact/exploitability:** A panel/app/worker/extension (any caller permitted by the vcs policy) issues `runtime.callMain("vcs.applyEdits", [{ edits: [{ kind: "write", path: "../../../home/werg/.bashrc", content: { kind: "text", text: "..." } }] }])` (or a `delete` op to remove arbitrary files via rmTolerant). The path is stored into GAD state unvalidated; when that state is materialized to disk — automatically during build-source graph discovery after the edit, or on context-fork/merge checkout — store.ts:492's `path.join(dir, "..", "..", ...)` writes/overwrites/deletes outside the intended worktree, on the host running the server, as the server OS user. This is a sandbox escape: a panel or workerd isolate gains arbitrary host filesystem write/overwrite/delete (e.g. clobber a config or shell rc, plant content, or delete files), bounded only by the server process's filesystem permissions. The harness toVcsPath guard does not protect this path because it is client-side and the server accepts raw paths.

- **Evidence:** RPC schema has no traversal guard — packages/shared/src/serviceSchemas/vcs.ts:50-77: `z.object({ kind: z.literal("write"), path: z.string(), content: vcsFileWriteContentSchema, mode: z.number().int().optional() })` (and delete/create/chmod likewise `path: z.string()`).
Handler forwards unmodified — src/server/services/vcsService.ts:228-236: `const result = await vcs.applyEdits({ head, baseStateHash, edits: input.edits, actor: {...} })`.
applyEdits keys by op.path verbatim — src/server/gadVcs/workspaceVcs.ts:1367-1371: `files.set(op.path, { path: op.path, contentHash: digest, mode: op.mode ?? files.get(op.path)?.mode ?? 33188 });` with no `..`/absolute check anywhere in the op loop (lines 1351-1387).
On-disk sink — src/server/gadVcs/store.ts:491-492: `const relPath = file.path; const absPath = path.join(dir, ...relPath.split("/"));` and delete loop store.ts:485: `await this.rmTolerant(path.join(dir, ...relPath.split("/")));`.
dir is the real head root — src/server/gadVcs/workspaceVcs.ts:286-289: `if (head === VCS_MAIN_HEAD) return this.deps.workspaceRoot; if (head.startsWith("ctx:")) return this.contextDir(head.slice(4));`.
Client-side-only guard that is bypassed — workspace/packages/harness/src/tools/tool-vcs.ts:20-22: `if (!abs.startsWith(root)) { throw new Error(\`Path escapes the workspace root: ${path}\`); }`.
Probe: node `path.join("/home/werg/natstack3/.contexts/ctxABC", ..."../../../tmp/PWNED.txt".split("/"))` => `/home/werg/tmp/PWNED.txt`.

- **Fix:** Validate/normalize every EditOp.path at the server boundary before it enters GAD state, and defense-in-depth at the materialize sink. In src/server/services/vcsService.ts applyEdits (and in WorkspaceVcs.applyEdits over input.edits), reject any path that, after `path.posix.normalize`, has a leading `/`, is absolute, contains a `..` segment, or whose normalized form starts with `../`; require relative, non-empty, no NUL. Concretely a guard like: `const n = p.replace(/\\/g, "/"); if (n === "" || n.startsWith("/") || n.split("/").some(seg => seg === ".." )) throw new Error("vcs path escapes worktree: " + p);` applied to op.path for every op. Additionally harden WorkspaceStore.materializeState/materializeFileList (store.ts:485,492,614): after computing absPath, assert `path.resolve(absPath).startsWith(path.resolve(dir) + path.sep)` and throw otherwise, so no state (even a pre-existing poisoned one) can ever write outside `dir`.

## #2 — Spectrolite drops edits in the 800ms debounce on file/vault switch
- **Verdict:** confirmed  | reported critical → assessed **high**
- **Why:** The finding is real and reachable in the current tree (docController.ts was NOT modified vs HEAD; the cited lines match content I verified). Full traced chain:

1. User keystroke -> Lexical update listener -> DocController.onUserEdit callback (registered in load(), docController.ts:167) -> scheduleCommit() (docController.ts:192). scheduleCommit sets an 800ms timer (quiescenceMs default 800, line 194). The edit at this point lives ONLY in Lexical editor state + the in-memory block registry's dirty marks; it has NOT been written to VCS. commitNow() (the only thing that calls vcs.applyEdits) has not run yet.

2. A file switch goes through app.openFile() (createApp.ts:134). It only mutates store state (activePath, recentPaths, etc.) and setStateArgs — it does NOT call commitActiveDoc()/commitNow(). commitActiveDoc is wired only into SendToScribe (flush-before-dispatch), nowhere on the switch path.

3. EditorPane renders <DocumentEditor key={activePath} .../> (EditorPane.tsx:103). Because relPath/activePath is the React key, changing the active file unmounts the old DocumentEditor and mounts a new one.

4. The old DocumentEditor's cleanup effect (DocumentEditor.tsx:105-113) runs controllerRef.current?.dispose() with NO commitNow() first, then nulls coreRef/undoRef.

5. DocController.dispose() (docController.ts:294-303) sets disposed=true and CLEARS the commit timer (clear(this.commitTimer)) without flushing. The pending, un-fired debounced commit is silently discarded, and the Lexical editor state is torn down on unmount, so the typed text is gone.

Vault switch is the same or worse: VaultController.selectVault/switchVault call reopen() (vaultController.ts:46,56) which reloads the whole panel — again no flush.

No safety net exists: grep found no beforeunload/visibilitychange/blur/pagehide flush anywhere in the panel, and no test covers dispose-time flushing.

I down-adjusted from critical to high: it is genuine silent data loss but bounded to edits made within the ~800ms quiescence window immediately before a switch, and a deliberate file/vault switch typically lands shortly after the user stops typing, so the realistic loss is the final fragment of typing rather than arbitrary amounts. Still a direct violation of the product's stated promise ('your changes save automatically and never get yanked away').

- **Impact/exploitability:** Not a security exploit — silent user data loss. Concrete scenario: user types a sentence at the end of a note, then within 800ms clicks another file in the FileTree/QuickOpen (or switches vault). openFile() flips activePath, EditorPane remounts DocumentEditor, the old controller's pending debounce timer is cleared in dispose() and Lexical state is destroyed. The last burst of typing was never sent to vcs.applyEdits and is permanently lost with no warning. Same on vault switch via reopen().

- **Evidence:** docController.ts:294-303 dispose() — clears the timer, never flushes:
```

    this.commitTimer = null;
  }
  this.offUserEdit?.();
  this.offHead?.();
}
```
docController.ts:192-202 scheduleCommit() — only an 800ms debounce holds the edit:
```

...
this.commitTimer = set(() => { this.commitTimer = null; void this.commitNow(); }, ms);
```
DocumentEditor.tsx:104-114 cleanup — dispose with no commit:
```
useEffect(() => {
  return () => {
    controllerRef.current?.dispose();
    controllerRef.current = null;
    ...
  };
}, [app, relPath]);
```
EditorPane.tsx:103 — relPath-keyed remount on switch: `<DocumentEditor key={activePath} relPath={activePath} ...>`
createApp.ts:134-148 openFile() — sets state only, no commitActiveDoc()/commitNow().

- **Fix:** Flush the pending debounced commit synchronously-initiated before tearing the timer down. Minimal fix in DocController.dispose(): if a commitTimer is pending, fire commitNow() before clearing — e.g. capture the pending state and call commitNow() (it is async but reads the still-alive editor state synchronously enough; better: make scheduleCommit track a `pending` flag and have dispose() do `if (this.commitTimer != null) { clear(this.commitTimer); this.commitTimer = null; void this.commitNow(); }` BEFORE setting disposed=true, since commitNow early-returns when disposed). Equally important: in DocumentEditor's cleanup, await/trigger a flush before nulling coreRef — call controllerRef.current?.commitNow() (or a new flushAndDispose()) prior to dispose so the editor is still readable. Note commitNow() guards on `this.disposed` (line 213), so ordering matters: flush must happen before disposed=true. Best design: add DocController.flushAndDispose() that runs commitNow() then dispose(), and call it from the unmount cleanup; also flush on vault reopen() before navigating.

## #4 — Conflicted publish to main bypasses beforeAdvance approval but materializes into main
- **Verdict:** confirmed  | reported High → assessed **high**
- **Why:** A panel/app caller's `vcs.publish` routes to `mergeHeads(VCS_MAIN_HEAD, sourceHead, { beforeAdvance })` (src/server/services/vcsService.ts:362), which dispatches to `mergeHeadsWithUnlockedApproval` because `beforeAdvance` is present. The approval gate (`mainAdvanceGate.approve`, wired via `mainAdvanceHook` at vcsService.ts:105-112) is meant to require workspace-repo-write capability/meta approval for any panel/app advance of main.

Tracing the conflicted case in workspaceVcs.ts:
- `mergeHeadsWithUnlockedApproval` (line 990) first does `commitHead(main, { beforeAdvance })` — but that only snapshots main's OWN uncommitted state; on a clean main it does not advance, so no approval fires.
- It then calls `prepareMergeAdvance` (line 1009). In the CLEAN branch (lines 1108-1132) this returns `kind:"candidate"` with an `event`, and the caller runs `beforeAdvance(prepared.event)` at line 1013 BEFORE `finalizeApprovedMerge` materializes — so clean publishes are gated.
- In the CONFLICTED branch (lines 1134-1166) `prepareMergeAdvance` itself stages the provisional tree, sets the pending merge, and calls `this.vcs.materializeState(staged.stateHash, targetDir)` at line 1151 — writing the source-side content + conflict markers into the MAIN workspace (`dirForHead("main")` = `workspaceRoot`) and a `.merge-conflicts` summary (syncConflictSummary, line 1157) — then returns `kind:"done"`. Back in the caller, line 1010 `if (prepared.kind === "done") return prepared.result;` returns immediately, so `beforeAdvance(...)` at line 1013 is NEVER reached. No approval for the conflicted materialization.

Worse than worktree-only dirtying: the pending conflicted merge auto-resolves into a RECORDED main advance with NO gate. `ensureFresh()` (workspaceVcs.ts:586/612) calls `commitHead(VCS_MAIN_HEAD, { actor: SYSTEM_ACTOR })` with NO beforeAdvance. The non-approval `commitHead` (line 326-365) detects the pending merge and snapshots with `force:true`, `parentStateHashes:[theirsStateHash]`, `eventKind:"state.merge_applied"`, transitionKind `merge-resolution`, clears the pending merge, and emits `state-advanced` — i.e. it commits the unapproved, conflict-marked source content as a real main merge. `ensureFresh()` is called automatically by the build system (buildV2/index.ts:246,388,508,534,712 — bindRuntimeImage/getBuild on any panel load or build trigger), so a panel/app caller need only trigger a conflicting publish and let routine build activity finalize the unapproved advance of main.

- **Impact/exploitability:** A panel/app userland caller (vcs.publish policy allows shell/panel/app) commits work on its context head that conflicts with main (e.g. edits a file main also changed), then calls vcs.publish. Because the merge conflicts, the approval gate that would have blocked a clean publish is skipped, yet the source-side content + conflict markers are written into the real main workspace tree. The pending merge is then auto-finalized into a recorded `state.merge_applied` advance of main by the next `ensureFresh()` — invoked automatically by routine build activity under SYSTEM_ACTOR with no gate. Net: a panel can advance main with arbitrary (conflict-marked) content while completely bypassing the workspace-repo-write capability / meta-change approval that protects clean publishes. This breaks the core invariant that non-shell callers cannot move main without explicit user approval, and can poison main with unreviewed code/markers that subsequently feed builds.

- **Evidence:** src/server/services/vcsService.ts:362 `const result = await vcs.mergeHeads(VCS_MAIN_HEAD, sourceHead, { actor..., ...mainAdvanceOptions(ctx, deps, { operation: "publish", sourceHead }) });`

workspaceVcs.ts:1010-1013 `if (prepared.kind === "done") return prepared.result;\n if (prepared.kind === "retry") continue;\n await opts.beforeAdvance(prepared.event);` — conflicted returns "done" at 1010, never reaching the beforeAdvance at 1013.

workspaceVcs.ts:1146-1157 (conflicted branch of prepareMergeAdvance): `setPendingMerge(... materialized:false)` → `await this.vcs.materializeState(staged.stateHash, targetDir)` (line 1151) → `setPendingMerge(... materialized:true)` → `await this.syncConflictSummary(targetHead)` — writes conflict tree into main with no approval.

workspaceVcs.ts:612 `const result = await this.commitHead(VCS_MAIN_HEAD, { summary: "workspace scan", actor: SYSTEM_ACTOR });` — ensureFresh resolves the pending merge with no beforeAdvance.

workspaceVcs.ts:354-365 (commitHead, non-approval path): `...(pending ? { force: true, parentStateHashes:[pending.theirsStateHash], eventKind:"state.merge_applied" } : {})` then `clearPendingMerge` + emit state-advanced — records the unapproved merge onto main.

- **Fix:** Run the approval gate for the conflicted publish path before any materialization onto main. In `prepareMergeAdvance`, do not materialize the provisional conflicted tree onto the target inside the locked section; instead return a `candidate`-like result carrying the provisional event so `mergeHeadsWithUnlockedApproval` invokes `beforeAdvance(event)` (line 1013) for BOTH clean and conflicted cases, and only materialize/set the pending merge after approval succeeds. Additionally, since the pending merge is later finalized by an ungated `ensureFresh()`/`commitHead` under SYSTEM_ACTOR, the approval must be obtained at merge-creation time (as above) — relying on the resolution commit to gate it is insufficient because the system-actor scan path has no beforeAdvance. Equivalently: gate the `materializeState` at workspaceVcs.ts:1151 (and the analogous plain mergeHeads conflicted branch at 979) on the same approval the clean candidate path uses when targetHead === VCS_MAIN_HEAD.

## #5 — Workspace-panel CDP navigation bypassable via frameId
- **Verdict:** confirmed  | reported High → assessed **high**
- **Why:** The guard `rejectWorkspacePanelNavigationCdp` in src/server/cdpBridge.ts is meant to stop raw CDP from navigating workspace panels (the server-authoritative "no-state-yank" rule: workspace panels must be navigated only via panelTree.navigate, not by a panel/agent yanking the top frame to an arbitrary URL via CDP). The guard explicitly carves out an exemption that the finding describes, and that exemption is unsound.

src/server/cdpBridge.ts:846-849 (current code, opened):
  if (!this.isWorkspaceTarget(targetId)) return false;
  if (msg.method === "Page.navigate" && typeof msg.params?.["frameId"] === "string") {
    return false;   // <-- does NOT reject
  }
The intent is clearly to permit sub-frame (iframe) navigation while blocking main-frame navigation, but the check only verifies that *some* string frameId is present — it never verifies the frameId refers to a child frame rather than the page's main frame. In CDP, Page.navigate({url, frameId}) navigates the named frame; if frameId is the main frame's id, it is a full top-level navigation of the workspace panel.

The exempted command is forwarded unchanged: handleClientConnection (cdpBridge.ts:727) calls the guard, and on `false` falls through to provider.send({type:"cdp:command", method, params,...}) at :740. On the host side, src/main/cdpHostProvider.ts:650 forwards it verbatim: `contents.debugger.sendCommand(method, params, sessionId)` — no re-validation of frameId. So the main-frame navigation reaches Chromium's real debugger.

The main frame id is trivially obtainable by the same client: only the 5 write methods (Page.navigate/reload/stopLoading/navigateToHistoryEntry/resetNavigationHistory) are in `navigationMethods` (cdpBridge.ts:850-857); read commands such as Page.getFrameTree / Page.enable+Page.frameNavigated / Page.getNavigationHistory are NOT blocked, so the attacker reads the main frameId, then resubmits Page.navigate with that frameId and slips past the guard.

- **Impact/exploitability:** A panel/agent that has an authenticated CDP client connection to a workspace target (CDP automation is now broadly enabled per commit 5c33791a "allow CDP automation for all panels") sends Page.enable + Page.getFrameTree, reads the main frame's frameId, then sends Page.navigate({url:"https://attacker.example/", frameId:"<mainFrameId>"}). The guard returns false (because a string frameId is present), the command is forwarded to Chromium, and the workspace panel's top frame is navigated to the attacker URL — exactly the state-yank/redirect the guard exists to prevent (phishing/UI-redress of a trusted workspace panel, escaping panelTree.navigate's server-authoritative routing). Precondition: the caller must already hold a valid CDP client connection to a workspace target.

- **Evidence:** src/server/cdpBridge.ts:846-849 — `if (!this.isWorkspaceTarget(targetId)) return false; if (msg.method === "Page.navigate" && typeof msg.params?.["frameId"] === "string") { return false; }` (returns false = "do not reject"; no check that frameId is a sub-frame). src/server/cdpBridge.ts:850-857 — only Page.navigate/reload/stopLoading/navigateToHistoryEntry/resetNavigationHistory are blocked; Page.getFrameTree and other read methods are allowed, so the main frameId is readable. src/server/cdpBridge.ts:727,740 — guard sits on the client→provider path and falls through to provider.send on `false`. src/main/cdpHostProvider.ts:650 — `contents.debugger.sendCommand(method, params, sessionId)` forwards params (incl. frameId) verbatim to Chromium with no validation.

- **Fix:** Reject Page.navigate for workspace targets unconditionally, OR validate the frame is non-main before allowing it. Minimal/safe fix: remove the frameId carve-out so workspace Page.navigate is always blocked — delete cdpBridge.ts:847-849. If sub-frame navigation must be supported, the bridge must resolve the target's main frameId (e.g. cache it from Page.getFrameTree / frameNavigated events for that target) and reject when `msg.params.frameId === mainFrameId` (and also reject when frameId is absent, since absent frameId also navigates the main frame).

## #9 — Publish conflicts not resolvable from the editor
- **Verdict:** confirmed  | reported High → assessed **high**
- **Why:** I traced the full publish-conflict flow in the current tree and the finding holds. A "Publish" is pull-main-then-publish: publishController.publish() calls vcs.merge("main") into the caller's own ctx head (publishController.ts:109). ## #11 — Pinned-head EV recompute uses main evMap/contentHashes for context changes
- **Verdict:** partially-confirmed  | reported Medium → assessed **low**
- **Why:** The structural defect is real and reachable. In stateTrigger.ts, `processPinnedHead` (the non-main head path, reached for ctx: heads taking the incremental, non-rediscovery branch) computes EVs from the MAIN baselines `this.graph`, `this.evMap`, `this.contentHashes`, mixing in only the changed units' fresh hashes from `event.stateHash` via `recomputeFromNodes(this.graph, changedNames, this.evMap, this.contentHashes, updated)`. Unlike the main `process` path, `processPinnedHead` never updates `this.*` and never persists, so it always recomputes from the pristine main baseline. The finding's two-step scenario is exactly the failure: (step1) ctx edits dependency D -> D's fresh hash flows in and D+P get recomputed correctly off main; (step2) ctx edits dependent P only -> recompute uses main's content hash for D (d0), NOT the ctx's d1, so P's EV omits the unpublished D change.

However the impact is NARROWER than "an artifact that omits the unpublished dependency change is reused for that head." Serving for a ctx ref does NOT use this eager EV at all: getBuild's ctx path (index.ts:455,474-476) does a full `computeEffectiveVersions(graphAtState, hashes)` at the exact ctx state, producing the CORRECT EV/buildKey, so the directly-served ctx head is always correct. Also `buildUnit`/`doBuild` materialize sources (incl. transitive deps) from `stateRef = event.stateHash` (builder.ts:1654-1658), so the eager artifact CONTENT is built from the correct ctx sources (D=d1). The only thing that is wrong is the buildKey under which that correct-content artifact is stored: f(P, hash(p1, depSig(D=d0))). That key is the LEGITIMATE buildKey of a different state where P=p1 and D=d0. Since buildStore is shared and purely content-addressed by buildKey, this enables cross-state cache poisoning: a later correct serve for a state that genuinely has P=p1/D=d0 can retrieve the ctx artifact built from D=d1 (wrong dep code), or conversely the ctx eager build is silently skipped/wasted. So: real bug, correctness (stale/divergent dependency code), but it manifests as a cross-state buildKey collision, not a mis-serve of the editing head itself, and requires a buildKey collision with another live state. Hence partially-confirmed at low severity.

- **Impact/exploitability:** Not directly exploitable as a mis-serve of the editing context head (getBuild always recomputes the correct full EV for ctx: refs). The concrete failure: on a ctx head, an agent edits dependency D (step1) then edits dependent P (step2) without re-touching D's files. processPinnedHead's step2 recompute yields P's EV as hash(content(P=p1), depSig(D=d0-from-main)), omitting the ctx's D=d1. The eager build of P (content correctly built from D=d1 ctx sources) is stored under the buildKey of the FALSE state (P=p1 with D=d0). Because buildStore is shared and content-addressed, if any other state legitimately has P=p1/D=d0 and is served, it either (a) retrieves this ctx artifact carrying the wrong D=d1 dependency code, or (b) the ctx eager build finds the other state's already-cached d0 artifact and skips — wasting the eager build. Net effect: stale/divergent dependency code can be served across contexts under a colliding buildKey, or eager warming is silently ineffective. Bounded: requires a buildKey collision between the stale-baseline EV and another live state's correct EV; the actively-edited head's own serve is unaffected.

- **Evidence:** src/server/buildV2/stateTrigger.ts:270 `const result = recomputeFromNodes(this.graph, changedNames, this.evMap, this.contentHashes, updated);` — `this.evMap`/`this.contentHashes`/`this.graph` are the MAIN-head baselines (set in constructor/`process`/`fullRediscovery`, never updated by `processPinnedHead`). stateTrigger.ts:260-268 only fetches fresh hashes for `changedNames` at `event.stateHash`, leaving every non-edited dependency at its main content hash. Contrast effectiveVersion.ts:108-136 `recomputeFromNodes`: `const newHashes = { ...contentHashes, ...updatedHashes }` then `buildDepSignatures(... newHashes)` — a dep not in `changedNames`/`updatedHashes` keeps the stale (main) `contentHashes[depName]`. Correct serving path index.ts:474-476 `const hashes = await contentHashesAt(graphAtState, buildState); const result = computeEffectiveVersions(graphAtState, hashes);` recomputes the full EV at the ctx state. builder.ts:1654-1658 materializes sources from `stateRef` (= `event.stateHash`), so eager artifact content reflects the ctx state, only the buildKey (stateTrigger.ts:409 `computeBuildUnitKey(node, ev)`) is computed from the stale-baseline `ev`.

- **Fix:** Compute pinned-head EVs from the pinned head's OWN baseline rather than main's. Minimal correct fix: in `processPinnedHead`, always discover the graph and hash all units at `event.stateHash` and call `computeEffectiveVersions(graph, contentHashesAtState)` (i.e. take the same path the `unmatched || manifestTouched` branch already takes at stateTrigger.ts:245-253, but unconditionally for the incremental case too), instead of `recomputeFromNodes(this.graph, ..., this.evMap, this.contentHashes, ...)`. This guarantees every dependency's content hash reflects the ctx state, so the eager buildKey equals the correct ctx buildKey that serving computes — eliminating both the wasted-build and the cross-state collision. (Alternatively, maintain a per-head baseline map keyed by head name and feed the prior ctx baseline into recomputeFromNodes; or simply drop eager builds for non-main heads and rely solely on the already-correct on-demand getBuild ctx path.)

- **Impact/exploitability:** No attack path. ## #7 — Publication envelope IDs omit origin head/hash → fork id collision
- **Verdict:** refuted  | reported High → assessed **none**
- **Why:** The finding claims the pub envelope id lacks a per-fork discriminator, so two forks publishing the "same" logical event collide on the channel log's UNIQUE (log_id, head, envelope_id). The pub id is in fact `pub:${event.envelopeId}:${target.channelId}` (gad-store/index.ts:1704), and the DESTINATION channelId is exactly the per-fork discriminator that the finding says is missing. Tracing the fork path: the fork worker always allocates a fresh forked channel `forkedChannelId = "fork:${opts.channelId}:${crypto.randomUUID().slice(0,8)}"` (fork/fork.ts:121) and clones the channel DO into it; there is NO mode that reuses the parent channelId. The forked agent's postClone re-subscribes to that NEW channelId (agent-vessel.ts:1855-1861), and the agent loop only ever publishes to its single `loop.channelId` (agent-loop-driver.ts:392, `publish: { channels: [{ channelId: loop.channelId }] }`), which after postClone equals the forked channelId. So every fork's publications land in a DISTINCT channel log_id, and the pub envelope_id carries that distinct channelId suffix. Moreover the UNIQUE constraint is scoped to (log_id, head, envelope_id): even if two forks happened to compute an identical pub id string, the rows live under different log_id (different channel), so no constraint violation. The forked channel's inherited (pre-fork) pub events carry the PARENT channelId suffix (`pub:<X>:<parentChannelId>`), while post-fork republications carry the new suffix (`pub:<X>:<forkedChannelId>`), so they cannot collide within the forked channel's lineage either. The only collision the constraint produces within a single channel is the INTENDED idempotency/dedup of re-publishing the same logical event to the same channel (replayed, not fatal — tested in gadStore.test.ts:252-291). No reachable path lets two forks collide.

- **Impact/exploitability:** No failure scenario. Forks never share a destination channel log_id, so the pub envelope_id (which embeds the destination channelId) is unique per fork, and the UNIQUE constraint — being scoped to (log_id, head, envelope_id) — cannot be violated across forks. The constraint only triggers for genuine same-channel idempotent re-publication, which is the intended dedup-by-id behavior and is handled as a non-fatal replay, not a fatal error.
